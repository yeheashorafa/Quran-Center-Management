# نقل الطالب بين الحلقات

## الهدف

تنقل العملية الطالب بين حلقتين داخل البرنامج الأساسي مع الإبقاء على:

- نفس `student_id`.
- ملف الطالب وبياناته.
- جلسات التسميع القديمة.
- الاختبارات الرسمية.
- تاريخ التسجيلات السابقة.
- سجل مستقل لعملية النقل.

## المسار

```text
POST /api/manager/students/:studentId/transfer
```

الصلاحية المطلوبة:

```text
transfers.manage
```

## جسم الطلب

```json
{
  "toHalaqaId": "UUID",
  "transferDate": "YYYY-MM-DD",
  "note": "ملاحظة اختيارية",
  "idempotencyKey": "معرف فريد للعملية"
}
```

`transferDate` هو تاريخ بدء التسجيل في الحلقة الجديدة. ينتهي التسجيل السابق في اليوم الذي يسبقه، لذلك لا يحدث تداخل بين الحلقتين في تقارير الحضور أو شاشة التسميع.

## العملية داخل Transaction

1. التحقق من الطالب والتسجيل الأساسي النشط.
2. التحقق من الحلقة الجديدة وأنها نشطة وفي البرنامج نفسه.
3. منع النقل إلى الحلقة الحالية.
4. منع النقل بتاريخ مستقبلي أو في تاريخ بداية التسجيل الحالي.
5. منع النقل الرجعي إذا وُجد تسميع أو اختبار في الحلقة القديمة في تاريخ النقل أو بعده.
6. إغلاق التسجيل السابق بحالة `TRANSFERRED`.
7. إنشاء تسجيل نشط جديد.
8. إنشاء سجل في `student_transfers`.
9. إنشاء `AuditLog` باسم `STUDENT_TRANSFERRED`.

أي فشل في خطوة يؤدي إلى `ROLLBACK` ولا يُترك الطالب بين حالتين ناقصتين.

## منع التكرار والتعارض

- كل طلب يحمل `idempotencyKey`.
- إعادة إرسال الطلب نفسه تعيد نتيجة العملية السابقة ولا تنشئ نقلاً جديداً.
- التنفيذ يستخدم مستوى العزل `Serializable`.
- تحديث التسجيل القديم مشروط ببقائه نشطاً، لذلك أي تغيير متزامن يعيد `409 Conflict`.

## واجهة المدير

داخل ملف الطالب تظهر:

- الحلقة الحالية.
- نموذج نقل إلى حلقة نشطة أخرى.
- تاريخ بدء الحلقة الجديدة.
- ملاحظة النقل.
- سجل جميع عمليات النقل مع المصدر والهدف والمنفذ والتاريخ.
