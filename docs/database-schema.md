# مخطط قاعدة البيانات – الإصدار الأول

## المبادئ المعتمدة

- `student_id` هو هوية الطالب الثابتة، ولا تتغير عند انتقاله.
- الحلقة والمرحلة والشيخ ليست جزءاً من هوية الطالب.
- يمكن للطالب امتلاك تسجيل نشط واحد داخل كل برنامج؛ وهذا يسمح تاريخياً بتسجيل أساسي وتسجيل موسمي للطالب نفسه.
- البرنامج الموسمي غير مفعّل في بيانات الإصدار الأول، لكن الـSchema تدعمه لحفظ بيانات المخيم القديمة.
- لا تُخزّن كلمات المرور كنص صريح؛ جدول `users` يحتوي `password_hash` فقط.
- العلاقات الأساسية تستخدم UUID وForeign Keys، مع منع الحذف المتسلسل للسجلات التاريخية المهمة.
- اليوم لا يُخزّن في جلسة التسميع؛ يُستخرج من `session_date` ويُتحقق منه مقابل جدول الحلقة.
- الحذف الطبيعي Soft Delete بواسطة `deleted_at`، بينما الجداول التاريخية مثل `audit_logs` لا تُحذف.

## الوحدات الرئيسية

### المستخدمون والصلاحيات

- `users`
- `roles`
- `permissions`
- `user_roles`
- `role_permissions`
- `auth_sessions`

الجلسات ستكون Tokens عشوائية داخل Cookies آمنة، ويُخزّن Hash التوكن فقط في `auth_sessions`.

### البرامج والمراحل والحلقات

- `programs`: البرنامج الأساسي أو برنامج موسمي.
- `stages`: البراعم، الأشبال، الناشئون.
- `stage_default_schedules`: الأيام الافتراضية لكل مرحلة.
- `halaqat`: الحلقة ككيان مستقل عن الشيخ.
- `halaqa_schedules`: أيام الحلقة مع فترات سريان تاريخية.
- `halaqa_staff_assignments`: تعيين الشيخ على الحلقة بتاريخ بداية ونهاية.

### الطلاب

- `students`: ملف الطالب الثابت.
- `student_enrollments`: تاريخ تسجيل الطالب في البرامج والحلقات.
- `student_transfers`: عملية النقل وربط التسجيل القديم بالجديد.

يوجد Partial Unique Index يمنع أكثر من تسجيل نشط للطالب داخل البرنامج نفسه، مع السماح بتسجيل موازٍ داخل برنامج موسمي مختلف.

### جلسات التسميع

- `memorization_sessions`: جلسة واحدة للحلقة في التاريخ نفسه.
- `session_record_items`: حالة كل طالب داخل الجلسة.
- `session_activities`: الحفظ الجديد أو المراجعة أو السرد.

القيد `UNIQUE (session_id, student_id)` يضمن أن تعديل طالب واحد لا ينشئ نسخة مكررة ولا يحذف بيانات بقية الطلاب.

### الاختبارات الرسمية

- `official_exams`
- `official_exam_scopes`

الاختبار يرتبط بالطالب بواسطة `student_id` وبالمختبر بواسطة `examiner_user_id`. نطاق الاختبار منفصل حتى يدعم جزءاً أو سورة أو آيات أو صفحات.

### سجل التعديلات

- `audit_logs`

يخزن منفذ العملية، نوعها، الكيان، القيم القديمة والجديدة، Request ID، الجهاز، IP وUser Agent. يجب ألا توضع كلمات المرور أو التوكنات داخل القيم المسجلة.

## قيود PostgreSQL المهمة

- تسجيل نشط واحد للطالب داخل كل برنامج.
- شيخ أساسي نشط واحد لكل حلقة.
- فترة مفتوحة واحدة لليوم نفسه في جدول الحلقة.
- تواريخ النهاية لا تسبق تواريخ البداية.
- العلامة الرسمية بين 0 و100.
- `program_id` في تسجيل الطالب يجب أن يطابق برنامج الحلقة المختارة.
- عملية النقل لا يمكن أن تربط التسجيل نفسه كمصدر وهدف.

## بيانات Seed

يضيف `prisma/seed.ts`:

- البرنامج الأساسي.
- المراحل الثلاث الدائمة.
- الأيام الافتراضية لكل مرحلة.
- الأدوار: الشيخ، مدير المركز، المختبر.
- مجموعة الصلاحيات الأساسية وربطها بالأدوار.

لا يضيف Seed مستخدمين أو كلمات مرور افتراضية.
